<h2 id="sec-0">0. 配置文件基础与类型推导</h2>

配置文件是常规的 Node.js 模块，运行在构建阶段与服务端，不会被打包到客户端浏览器中。

<h3 id="sec-0-1">0.1 配置文件格式</h3>

根据项目规范，支持以下三种主要格式：

1. **TypeScript 格式**（<Icode>next.config.ts</Icode>）— 最新版官方推荐，提供完美的类型安全与 IDE 自动补全。
2. **ESM 格式**（<Icode>next.config.mjs</Icode>）
3. **CommonJS 格式**（<Icode>next.config.js</Icode>）

<h3 id="sec-0-2">0.2 基础配置写法</h3>

#### TypeScript 规范

```ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  /* 具体的配置项 */
};

export default nextConfig;
```

#### 函数式配置

根据不同的构建阶段（如开发服务器、生产构建等）加载不同的配置：

```mjs
import { PHASE_DEVELOPMENT_SERVER } from 'next/constants.js';

export default (phase, { defaultConfig }) => {
  if (phase === PHASE_DEVELOPMENT_SERVER) {
    return {
      /* 开发环境专属配置 */
    };
  }

  return {
    /* 其他环境配置 */
  };
};
```

<Icode>phase</Icode> 包含：<Icode>phase-export</Icode>（导出阶段）、<Icode>phase-production-build</Icode>（生产构建）、<Icode>phase-production-server</Icode>（生产服务）、<Icode>phase-development-server</Icode>（开发服务）以及 <Icode>phase-test</Icode>（测试阶段）。

<h2 id="sec-1">1. 路由与请求级配置</h2>

路由与请求级配置（<Icode>headers</Icode>, <Icode>redirects</Icode>, <Icode>rewrites</Icode>）在文件系统检查（包括 <Icode>public</Icode> 静态资源和页面路由）之前或期间触发，是控制流量的重要手段。

<Tip>
**测试提示**：自 Next.js 15 起，支持在单元测试中通过导入 <Icode>next/experimental/testing/server</Icode> 中的 <Icode>unstable_getResponseFromNextConfig</Icode> 来直接对这些自定义路由逻辑进行单元测试，极大方便了复杂重写与重定向逻辑的回归验证。
</Tip>

<h3 id="sec-1-1">1.1 自定义响应标头 (headers)</h3>

用于为匹配的请求路径注入自定义 HTTP 响应头。

```js
// next.config.js
module.exports = {
  async headers() {
    return [
      {
        source: '/about',
        headers: [
          {
            key: 'x-custom-header',
            value: 'my custom header value',
          },
          {
            key: 'x-another-custom-header',
            value: 'my other custom header value',
          },
        ],
      },
    ];
  },
};
```

#### 关键匹配属性

每个匹配对象除了 <Icode>source</Icode> 和 <Icode>headers</Icode> 外，还可以设置：<Icode>basePath</Icode>（设为 <Icode>false</Icode> 时不包含全局前缀）、<Icode>locale</Icode>（设为 <Icode>false</Icode> 时不自动添加 i18n 语言前缀）、以及 <Icode>has</Icode> / <Icode>missing</Icode> 条件化匹配（详见 1.4 节）。

#### 路径匹配规则

<Icode>source</Icode> 属性支持三种匹配模式：

1. **普通路径参数**：<Icode>/blog/:slug</Icode> 匹配 <Icode>/blog/hello-world</Icode>（单层路径）
2. **通配符参数**：<Icode>/blog/:slug*</Icode> 匹配 <Icode>/blog/a/b/c/d/hello-world</Icode>
3. **正则表达式参数**：<Icode>/blog/:post(\d&#123;1,&#125;)</Icode> 仅匹配数字型 ID（如 <Icode>/blog/123</Icode>）

#### 覆盖机制

如果多个配置匹配了相同的路径且存在相同的 Header Key，则**数组中排在后面的配置会覆盖前面的配置**；若 Header Key 不冲突，则会叠加应用。

<h3 id="sec-1-2">1.2 路由重定向 (redirects)</h3>

将请求重定向到另一个目标路径，并向浏览器返回相应的状态码。

```js
module.exports = {
  async redirects() {
    return [
      {
        source: '/old-blog/:slug',
        destination: '/blog/:slug',
        permanent: true,
        // true → 308 永久重定向；false → 307 临时重定向
      },
    ];
  },
};
```

#### 为什么使用 307 和 308？

传统的 302（临时）和 301（永久）重定向在许多旧浏览器中，会将原本的非 GET 请求（如 <Icode>POST</Icode>）重定向为 <Icode>GET</Icode> 请求。Next.js 强制使用 **307 临时重定向**和 **308 永久重定向**，可以明确指示浏览器保留原本的请求方法。

默认情况下，重定向规则**不会**应用于客户端路由导航（如 <Icode>&lt;Link&gt;</Icode> 或 <Icode>router.push</Icode>），除非通过 Middleware 实现了路径过滤。

<h3 id="sec-1-3">1.3 路由重写与代理 (rewrites)</h3>

重写相当于扮演了 **URL 代理** 的角色——浏览器地址栏 URL 保持不变，但服务端实际渲染的是目标路由的逻辑。在客户端导航中同样生效。

#### 单一数组形式

```js
module.exports = {
  async rewrites() {
    return [
      {
        source: '/about',
        destination: '/',
      },
    ];
  },
};
```

#### 细粒度三阶段重写

```js
module.exports = {
  async rewrites() {
    return {
      beforeFiles: [
        // 在检查 public 静态文件和非动态页面之前触发
        { source: '/some-page', destination: '/somewhere-else' },
      ],
      afterFiles: [
        // 在 public 静态文件检查之后，但动态路由检查之前触发
        { source: '/non-existent', destination: '/somewhere-else' },
      ],
      fallback: [
        // 在静态文件与动态路由都未匹配，即将渲染 404 之前触发
        { source: '/:path*', destination: 'https://my-old-site.com/:path*' },
      ],
    };
  },
};
```

#### 路由检查完整顺序

1. <Icode>headers</Icode>
2. <Icode>redirects</Icode>
3. <Icode>beforeFiles</Icode> 重写
4. <Icode>public/</Icode> 静态文件、<Icode>_next/static</Icode>、非动态页面
5. <Icode>afterFiles</Icode> 重写
6. 动态路由检验
7. <Icode>fallback</Icode> 重写（可用于增量迁移旧网站至 Next.js）
8. 渲染 404 页面

<h3 id="sec-1-4">1.4 条件匹配限制 (has 与 missing)</h3>

<Icode>headers</Icode>、<Icode>redirects</Icode>、<Icode>rewrites</Icode> 均支持 <Icode>has</Icode>（必须匹配）和 <Icode>missing</Icode>（必须不匹配）参数。支持基于请求头、Cookie、主机名、查询参数进行多维度过滤。

```js
module.exports = {
  async rewrites() {
    return [
      {
        source: '/:path*',
        has: [
          // Host 为 'example.com' 且 Cookie 中 authorized=true 时应用
          { type: 'host', value: 'example.com' },
          { type: 'cookie', key: 'authorized', value: 'true' },
        ],
        destination: '/another-page',
      },
      {
        source: '/:path*',
        has: [
          // 支持正则命名捕获，在 destination 中使用捕获变量
          { type: 'header', key: 'x-authorized', value: '(?<auth>yes|true)' }
        ],
        destination: '/home?auth=:auth',
      }
    ];
  }
};
```

<h2 id="sec-2">2. 工程构建与模块打包配置</h2>

在 Next.js 16.2.9 中，工程编译与外部包的依赖处理方案变得更加直接与成熟。

<h3 id="sec-2-1">2.1 服务端不打包外部包 (serverExternalPackages)</h3>

Next.js 默认会自动打包 Server Components 和 Route Handlers 中的所有依赖项。然而，对于包含原生二进制文件（如 <Icode>bcrypt</Icode>、<Icode>sharp</Icode>）或底层 Node.js 特有组件的库，打包往往会引起构建或运行期错误。

<Tip type="important">
**新版变动**：原有的实验性选项 <Icode>experimental.serverComponentsExternalPackages</Icode> 在最新版中已升级为顶层的正式配置项 <Icode>serverExternalPackages</Icode>。
</Tip>

```ts
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  serverExternalPackages: ['@acme/ui', 'some-native-package'],
};

export default nextConfig;
```

配置后，这些包将被排除在 Webpack/Turbopack 打包之外，改为在运行时通过原生 Node.js 的 <Icode>require</Icode> 动态加载。

<h3 id="sec-2-2">2.2 本地包自动转译 (transpilePackages)</h3>

当引用本地 Monorepos 模块或外部尚未被转译成 ES5/ES6 兼容代码的包时，Next.js 会自动使用内置编译器进行转译。不再需要社区的 <Icode>next-transpile-modules</Icode> 插件。

```js
module.exports = {
  transpilePackages: ['@my-monorepo/ui', 'lodash-es'],
};
```

<h3 id="sec-2-3">2.3 导入优化 (optimizePackageImports)</h3>

对于包含海量导出子模块的第三方包（例如 <Icode>@mui/material</Icode> 或 <Icode>date-fns</Icode>），通过该选项 Next.js 只加载您代码中实际引用的模块，从而加快热更新与构建速度。

```js
module.exports = {
  experimental: {
    optimizePackageImports: ['my-giant-ui-library'],
  },
};
```

像 <Icode>@mui/icons-material</Icode>、<Icode>lucide-react</Icode>、<Icode>lodash</Icode> 等常见库，Next.js 内部已默认进行了此类优化，无需重复声明。

<h3 id="sec-2-4">2.4 自定义 Webpack 配置 (webpack)</h3>

若内置的打包策略无法满足需求，可以通过 <Icode>webpack</Icode> 函数扩展底层配置：

```js
module.exports = {
  webpack: (config, { buildId, dev, isServer, nextRuntime, webpack }) => {
    // 区分服务端和客户端编译
    if (isServer) {
      config.plugins.push(
        new webpack.DefinePlugin({ __SERVER__: true })
      );
    }
    return config;
  },
};
```

<h3 id="sec-2-5">2.5 编译质量与编译目录配置</h3>

* <Icode>distDir</Icode> — 自定义打包产物输出文件夹，默认是 <Icode>.next</Icode>，必须在项目目录内。
* <Icode>eslint.ignoreDuringBuilds</Icode> — 设为 <Icode>true</Icode> 可在生产构建时忽略 ESLint 报错。
* <Icode>typescript.ignoreBuildErrors</Icode> — 设为 <Icode>true</Icode> 可在生产构建时忽略 TypeScript 类型检查错误。

<h2 id="sec-3">3. 图像优化配置 (images)</h2>

在 <Icode>next/image</Icode> 进行图像优化时，安全隔离至关重要。

<h3 id="sec-3-1">3.1 严格的外部域名匹配 (remotePatterns)</h3>

<Tip type="warning">
**安全警示**：旧有的 <Icode>domains</Icode> 字段在最新版中已被**废弃且移除**。您必须使用更加精确的 <Icode>remotePatterns</Icode> 进行匹配，以防应用被攻击者用作恶意图片的优化代理。
</Tip>

```ts
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '**.example.com',  // 匹配所有子域名
        port: '',
        pathname: '/images/**',      // 严格限制路径
      },
    ],
  },
};
```

<h3 id="sec-3-2">3.2 图像优化加速器 sharp</h3>

在最新版 Next.js 中，服务端进行高效率图片优化时，底层会自动识别并集成 **sharp** 引擎进行快速压缩，开发人员**无需在生产环境中手动安装 sharp 库**，这使得跨平台容器化部署更加顺畅。

<h2 id="sec-4">4. 缓存、渲染模式与最新部署实践</h2>

Next.js 16.2.9 在服务端缓存以及自托管方面提供了目前最先进的架构支持。

<h3 id="sec-4-1">4.1 新版缓存处理器 (cacheHandlers)</h3>

随着 <Icode>"use cache"</Icode> 缓存在最新版中的正式落地，Next.js 的数据与组件缓存机制经历了重大革新。

<Tip type="important">
**新版变动**：在以前版本中，自定义缓存使用 <Icode>experimental.incrementalCacheHandlerPath</Icode>。而在最新的 Next.js 16 中，已标准化升级为顶层的 <Icode>cacheHandlers</Icode> 映射关系，用以管理全局或分模块的缓存持久化方案。
</Tip>

```ts
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  cacheHandlers: {
    default: require.resolve('./cache-handler.js'),
  },
};

export default nextConfig;
```

#### 缓存类结构示例

```js
// cache-handler.js
const redisClient = require('./redis-client');

module.exports = class CacheHandler {
  constructor(options) {
    this.options = options;
  }

  async get(key) {
    const data = await redisClient.get(key);
    return data ? JSON.parse(data) : null;
  }

  async set(key, value, ttl) {
    await redisClient.set(key, JSON.stringify(value), 'EX', ttl);
  }
};
```

<h3 id="sec-4-2">4.2 开启新版缓存组件 (cacheComponents)</h3>

在最新 Next.js 16.2.9 中，<Icode>cacheComponents</Icode> 配置项用于激活最新架构的组件缓存指令（<Icode>"use cache"</Icode> 缓存指令支持）。

```ts
// next.config.ts
const nextConfig = {
  cacheComponents: true,
};
export default nextConfig;
```

<Tip type="important">
**版本演进对比：局部预渲染 (PPR)**

* **15.x 时代**：主要采用 `experimental: { ppr: true }` 开启，并在路由页面中定义 `export const experimental_ppr = true`
* **16.x 时代**：推荐全面启用 `cacheComponents` 模式，在细粒度组件或数据获取上通过 `"use cache"` 实现灵活的数据、组件缓存与静态切片隔离
</Tip>

<h3 id="sec-4-3">4.3 Standalone 独立构建部署 (output)</h3>

对于容器化（Docker）等自托管场景，推荐使用 <Icode>output: 'standalone'</Icode>。

```js
module.exports = {
  output: 'standalone',
};
```

在执行 <Icode>next build</Icode> 时，Next.js 会自动追踪并拷贝生产环境必需的所有依赖文件到 <Icode>.next/standalone/</Icode> 目录下。这允许您在目标服务器上**完全不需要安装 node_modules** 即可通过 <Icode>node server.js</Icode> 启动运行，极大地缩减了 Docker 镜像的体积。

<h2 id="sec-5">5. 网络、路径与其它基础配置</h2>

<h3 id="sec-5-1">5.1 资源与路径前缀</h3>

<Icode>assetPrefix</Icode> — 静态资源 CDN 前缀。在生产环境下，可以将 JS/CSS 静态资源的获取地址代理到外部 CDN 上。

```js
const isProd = process.env.NODE_ENV === 'production';
module.exports = {
  assetPrefix: isProd ? 'https://cdn.my-site.com' : undefined,
};
```

注意：只有 <Icode>.next/static/</Icode> 中的构建产物应该上传至 CDN。<Icode>public/</Icode> 目录下的静态资源依然通过本地服务器访问，<Icode>assetPrefix</Icode> 对它们不生效。

<Icode>basePath</Icode> — 应用路由的前缀路径。例如配置为 <Icode>/docs</Icode>，则原本路由 <Icode>/about</Icode> 自动映射为 <Icode>/docs/about</Icode>，所有 <Icode>&lt;Link&gt;</Icode> 和 <Icode>router</Icode> 路由导航均会自动附加前缀。

```js
module.exports = {
  basePath: '/docs',
};
```

<h3 id="sec-5-2">5.2 其它工程控制项</h3>

* <Icode>reactStrictMode</Icode> — 严格模式（App Router 模式下最新版默认启用），有助于捕捉非安全生命周期。
* <Icode>poweredByHeader</Icode> — 默认在响应头带上 <Icode>x-powered-by: Next.js</Icode>，设为 <Icode>false</Icode> 可移除以保障安全性。
* <Icode>trailingSlash</Icode> — URL 尾部斜杠控制。设为 <Icode>true</Icode> 时自动将 <Icode>/about</Icode> 重定向到 <Icode>/about/</Icode>。
* <Icode>productionBrowserSourceMaps</Icode> — 设为 <Icode>true</Icode> 将输出生产 SourceMap。通常建议保持 <Icode>false</Icode> 以防源码泄露。

<h2 id="sec-6">6. 开发调试与性能指标</h2>

<h3 id="sec-6-1">6.1 开发日志控制 (logging)</h3>

配置开发环境下数据获取（Fetch）的详细控制台日志。

```js
module.exports = {
  logging: {
    fetches: {
      fullUrl: true, // 打印 fetch 请求的完整 URL
    },
  },
};
```

<h3 id="sec-6-2">6.2 开发指示器 (devIndicators)</h3>

在开发调试阶段，编译状态指示器以及 ISR/页面的静态渲染指示器。在 Next.js 16 中可通过 <Icode>position</Icode> 控制其显示位置：

```js
module.exports = {
  devIndicators: {
    position: 'bottom-right',
    // 可选 'bottom-left', 'top-right', 'top-left'
  },
};
```

<h3 id="sec-6-3">6.3 页面性能归因 (webVitalsAttribution)</h3>

在采集 Web 性能指标时，可指定获取深层元素归因（如 CLS 最大布局偏移、LCP 最大内容渲染时间等），以精细化定位性能瓶颈：

```js
module.exports = {
  experimental: {
    webVitalsAttribution: ['CLS', 'LCP'],
  },
};
```

<h2 id="sec-7">7. 版本演进与废弃总结对照表</h2>

| 旧版配置 | 16.2.9 最新标准 | 说明与迁移建议 |
| :--- | :--- | :--- |
| `i18n (i18n.locales)` | 由 Middleware 替代 | Pages Router 下保留，App Router 下已被 Middleware 级动态路由完全接管 |
| `experimental.serverComponentsExternalPackages` | `serverExternalPackages` | 已标准化，直接作为顶层配置使用 |
| `experimental.incrementalCacheHandlerPath` | `cacheHandlers` | 演进为顶层多处理器映射关系 |
| `experimental.ppr` | `cacheComponents: true` | 实验性 PPR 演进为 "use cache" 组件级缓存模型 |
| `experimental.turbo` | `turbopack` | 提升为顶层配置项，支持更丰富的 alias 与 rules |
| `images.domains` | `images.remotePatterns` | 已彻底移除，出于安全考虑必须使用 remotePatterns |
| `AMP 支持` | 彻底移除 | 架构升级，不再支持 AMP 组件 |
| `next lint` | 从构建核心移除 | 建议在 CI 或 Pre-commit 阶段独立运行 |

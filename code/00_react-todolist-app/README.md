# 看板式待办事项管理系统 (Kanban Todo List)

本项目是针对前端机试面试题（要求包含：状态流转、标题/描述信息、响应式兼容移动端、以及弹窗新增/编辑功能）的高标准实现方案。通过严谨的组件拆分、CSS 模块化设计、以及 React 最佳实践，全方位展示前端开发的编码规范与架构设计能力。

## 🎯 面试核心考察点

在前端面试的机试环节中，此类题目表面看似简单，但考官通常会从以下几个维度深度评估候选人的专业水平：

1. **组件化设计与 React 思维**：
   - 状态设计是否合理？（例如：单向数据流、状态提升，避免冗余状态导致的“单源不一致”问题）。
   - 组件职责是否单一？（如：`Column` 仅负责列渲染与空状态，`TodoCard` 负责单项展示，`TodoModal` 负责交互）。
2. **移动端适配与用户体验 (UX)**：
   - 移动端不能直接平铺三栏（会导致排版过窄或横向滚动条体验极差），也不能生硬堆叠。
   - 考察是否能优雅地使用 Tab 栏在移动端切换视图，且保持高度的视觉一致性与平滑动效。
3. **副作用管理与生命周期**：
   - 弹窗的开启/关闭、数据回显时，是否会出现不必要的副作用？
   - 是否存在 React 19 禁止的“在 `useEffect` 中同步调用 `setState` 导致级联渲染”的性能与规范问题。
4. **CSS 架构与防污染**：
   - 是否使用了 CSS Modules、CSS 变量来维持干净的样式树？
   - 样式重置 (CSS Reset) 是否合理，排版是否现代优雅（如字重、圆角、阴影）。
5. **边缘情况 (Edge Cases) 与安全性**：
   - 输入校验（标题不能为空、字符长度限制）。
   - 数据溢出防暴撑（英文长连续字符强制折行 `word-break: break-word`）。
   - React 安全性（不使用 `dangerouslySetInnerHTML` 以防止 XSS 攻击）。
   - 可访问性 (a11y) 支持（键盘 `Esc` 退出弹窗，聚焦态高亮等）。

## 🛠️ 解决思路与技术实现

### 1. 状态流转设计 (State Management)
* **核心数据结构**：
  ```typescript
  interface Todo {
    id: string;        // 唯一标识符
    title: string;     // 任务标题 (必填)
    description: string; // 任务描述 (选填)
    status: 'pending' | 'in-progress' | 'completed'; // 状态流转
    createdAt: string; // 创建时间 ISOString
  }
  ```
* **单向数据流与状态提升**：任务的主状态保存在最顶层的 `KanbanBoard` 组件中，使用自定义的 `useLocalStorage` 进行状态持久化。各列（`Column`）与卡片（`TodoCard`）只接收数据和流转回调，保持纯净。

### 2. 弹窗的条件渲染与生命周期优化 (Vite/ESLint 零警告)
* 原本在弹窗中通过 `useEffect` 监听 `isOpen` 来初始化或重置输入框状态，但在 React 中这极易引发级联渲染性能警告。
* **最佳实践解决**：在 `KanbanBoard` 中使用条件渲染 `{isModalOpen && <TodoModal ... />}`。
  - 弹窗关闭时彻底卸载 (Unmount)，所有内部 state 自动重置销毁。
  - 打开时重新挂载 (Mount)，根据 `initialData` 进行纯净的初始渲染，并在 `useEffect` 内仅执行**聚焦**和**锁定背景滚动**等真实的 DOM 挂载副作用，完美契合 React 生命周期。

### 3. 移动端适配方案
* 使用媒体查询 `@media (max-width: 768px)`。
* 桌面端使用三列栅格（Grid）并排，具有充裕的留白与鼠标悬浮微悬浮阴影特效。
* 移动端下，多列网格折叠为 `display: none`，取而代之的是顶部轻量级的 Tab 导航条。用户点击 Tab 进行列切换，激活的列伴随 `@keyframes slideIn` 左右滑入，过渡极其流畅自然。

## 💡 后续可扩展功能与思路

本系统在底层设计上预留了高度解耦的接口，如需进一步丰富面试作品，可快速进行以下扩展：

### 1. 拖拽排序与流转 (Drag & Drop)
* **思路**：引入轻量级拖拽库（如 `@hello-pangea/dnd` 或原生 HTML5 Drag and Drop API）。
* **实现**：为 `Column` 添加 `dragover` 事件，`TodoCard` 标记为 `draggable`。在 `dragend` 时，获取当前卡片 `id` 和悬停的 `columnId`，调用 `handleMoveTodo` 即可完成流转，配合平滑的过渡动画将极大提升质感。

### 2. 过滤与全局搜索 (Search & Filters)
* **思路**：在 `KanbanBoard` 头部添加搜索输入框和状态/优先级筛选条件。
* **实现**：
  - 新增 `searchQuery` 状态。
  - 在 `useMemo(..., [todos, searchQuery])` 内部对数据进行二次过滤：
    ```javascript
    todos.filter(todo => todo.title.toLowerCase().includes(searchQuery.toLowerCase()))
    ```
  - 看板列直接显示过滤后的结果，做到搜索秒级响应。

### 3. 任务分类标签 (Tags / Labels)
* **思路**：为任务增加多维度属性，丰富展示。
* **实现**：
  - 在 `Todo` 数据结构中增加 `tags: string[]` 或 `priority: 'low' | 'medium' | 'high'`。
  - 在 `TodoModal` 中添加标签选择器（如 Select 或 Checkbox 组）。
  - 在 `TodoCard` 上渲染精美的小胶囊标签（不同优先级搭配不同颜色微渐变），提升视觉丰富度。

### 4. 任务时间期限与超期提醒 (Due Dates)
* **思路**：记录任务期限，并提供直观的警示。
* **实现**：
  - 数据模型增加 `dueDate: string`。
  - 卡片端比对当前时间，若接近到期日则展示橘黄色警告；若已超期则卡片背景变为淡红色，并在完成时展示“已按时完成”的徽章。





### ⏱️ 实战 1 小时时间拆解策略

#### **第一阶段：需求梳理与数据设计（0 - 10 分钟）**

- **不要急着写样式，先定数据结构**。在脑海中或草稿纸上写出 Todo 数据模型（`id`, `title`, `description`, `status`）。
- **确立状态中心**：决定把状态放在最顶层组件（如 `App.jsx` 或 `KanbanBoard.jsx`）。
- 快速写出状态流转逻辑（ Pending $\left right arrow$ In progress $\left right arrow$ Completed ）。

#### **第二阶段：核心功能实现（10 - 35 分钟）**

这是拿分的基本盘，必须优先保证功能闭环：

1. **基础布局**：用一个简单的 Flex 布局或 CSS Grid 把三个列表并排排开。
2. **列表渲染**：用 `.filter()` 过滤出不同状态的任务并渲染。
3. **状态流转**：写好移入、移出、完成的按钮，绑定状态修改函数。
4. **新增功能（简化版）**：
   - *省时技巧*：**第一步甚至可以用 `window.prompt` 录入数据**，确保数据通道是通的。
   - 确认没问题后，再花 5-10 分钟写一个简单的绝对定位 `div` 作为 Modal 遮罩（此时不需要写 Portal，也不用管 Escape 键绑定，先保证能提交表单）。

#### **第三阶段：移动端适配与持久化（35 - 50 分钟）**

1. **极简移动端适配**：
   - *省时技巧*：在 1 小时内，**不要挑战做我们刚才实现的 Tab 切换系统**。最快且合理的适配方案是用媒体查询让三栏在手机上**纵向单列堆叠** (`flex-direction: column`)，这已经符合了“兼容移动端”的及格线。
2. **本地存储**：
   - 快速在挂载时 `localStorage.getItem`，更新时 `localStorage.setItem`，没时间就不用封装复杂的自定义 Hooks。

#### **第四阶段：异常排查与微调（50 - 60 分钟）**

1. **输入校验**：拦截空标题。
2. **控制台检查**：打开 DevTools 看看有没有明显的 `key` 缺失报错。
3. **删除操作**：直接使用原生 `window.confirm('确认删除？')`，**在 1 小时内完全没必要手写复杂的 ConfirmDialog 弹窗**。

### 🔍 考官在 1 小时内到底看重什么？

机试时间紧迫，考官并非要求候选人成为“打字机”，他们真正评估的是你的**职业编码素养**：

1. **状态更新的不可变性 (Immutability)**：
   - 更新状态时，是用 `setTodos(prev => [...prev, newTodo])` 和 `.map()` / `.filter()`，还是错误地使用了 `todos.push()` 后强行更新？（**这是 React 面试的一票否决点**）。
2. **解耦意识**：
   - 哪怕组件写得简单，是否把 `TodoCard` 独立了出来？Props 传递是否清晰？
3. **代码整洁度**：
   - 命名是否具有自解释性（如 `handleMoveTodo` 而不是 `change`）。
   - 有没有基本的缩进，逻辑是否清晰。
4. **防御性编程**：
   - 用户什么都不输入点击保存时，程序会不会报错崩溃？输入超长字符时，布局有没有完全乱掉？

### 💡 快速通关的“作弊”小技巧

如果你在面试中遇到了类似的机试，可以使用以下技巧快速提效：

- **使用 HTML5 原生 `<dialog>` 标签**： 在 React 中，直接使用 `<dialog ref={dialogRef}>`。它原生支持 `showModal()`、背景遮罩层（通过 `::backdrop` 伪元素控制）、自动聚焦、以及**按 `Esc` 键自动关闭**。这可以帮你省去手写 Portal、绑定键盘事件和遮罩防冒泡的 15-20 分钟时间！
- **单文件 CSS 优先**： 没必要在机试中建立十几个 CSS Modules 文件。直接用一个全局的 CSS 文件（比如我们在 `index.css` 里设计的变量和通用类），或者在一个文件中用原生的 CSS 选择器，能极大节省在文件之间来回切换和引用的时间。

**总结**： 我们之前做出的成品是一个**“可供考官打 100 分”**的范例。在真实的 1 小时机试中，你只需要做到这个成品的 **50% - 60%**（即：纵向堆叠适配、简单绝对定位 Modal、原生 confirm 确认、基本 LocalStorage），就已经足够通过筛选，拿到优秀评级了！
